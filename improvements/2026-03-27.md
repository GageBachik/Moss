## Proposal 1: Add xcodebuild SPM + Signing Learnings
**Category:** failures
**Evidence:** Eval was stuck for 7-13 hours on snap-split and nospend-streak. Root cause: using `-target` flag which doesn't resolve SPM packages, and missing signing overrides for simulator. Discovered during escalation resolution at 15:14 UTC. Already noted in orchestrator-state.json knownIssues but not in learnings/design-build.md.
**Proposed Change:**
  File: /Users/runemoss/moss/learnings/design-build.md
  Change: Add to "Build Notes" section:
  ```
  - Always use -scheme (not -target) for xcodebuild when project has SPM dependencies -- -target skips package resolution
  - For simulator eval builds, always pass: CODE_SIGN_IDENTITY='-' CODE_SIGNING_REQUIRED=NO CODE_SIGNING_ALLOWED=NO
  ```
**Classification:** auto-apply
**Rationale:** These are hard-won build fixes that wasted 7+ hours. Encoding them in learnings prevents repeat failures for every future eval build.

**Category:** wins
**Evidence:** 6/10 concepts killed today all shared 3 patterns: (1) 8-12+ competitors, (2) MRR <$500/mo, (3) dominant incumbent with PMF. Meanwhile, all 4 validated concepts had <10 competitors, clear unmet need, and MRR $500+. This pattern is consistent enough to encode as a scout pre-filter.
**Proposed Change:**
  File: /Users/runemoss/moss/learnings/scout.md
  Change: Add to "Patterns That Fail" section:
  ```
  - Concepts where the niche already has 8+ iOS apps (even if they're mediocre) -- validator will kill these
  - Concepts targeting spending-averse audiences (budget trackers, pan tracking) -- monetization math fails
  ```
  Add to "Patterns That Work" section:
  ```

  - Apps riding TikTok wellness/lifestyle trends with 100M+ views but no dedicated app
  ```
**Classification:** auto-apply
**Rationale:** Scout generating concepts that validator consistently kills wastes pipeline cycles. Pre-filtering based on proven kill patterns improves throughput.

**Classification:** suggest
